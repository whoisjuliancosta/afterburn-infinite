3. **Overheal → new heart containers** (run.js `applyGem`): red progress accrues exactly as now (epsilon guard stays). When a heart completes and `ship.hp >= ship.maxHp`: `ship.maxHp += 1; ship.hp = ship.maxHp` and return `{kind:'red', healed:true, grown:true}`. The `full` (wasted) outcome is REMOVED — red gems are never wasted. No cap on maxHp ("and so on"). `grown:true` gets a distinct floater: `+1 MAX ♥`.
4. **HUD hearts row** (hud.js `drawHud`):
   - When `ship.hp === ship.maxHp && run.heartProgress > 0`, draw ONE extra empty heart container at the row's end — the in-progress next heart — with a partial fill showing `heartProgress` (clip the full-heart sprite bottom-up by the progress fraction over the empty sprite; source-rect drawImage, no new per-frame canvases). `drawHud` already receives `run` — read `run.heartProgress` from it.

5. **Interactions to preserve**: adrenaline threshold uses `hp/maxHp < 0.5` — grown maxHp naturally raises the adrenaline band; that's accepted behavior, note it in a test. Boss-kill heal (`main.js` killEnemy) stays `Math.min(ship.maxHp, hp+1)` — boss heal does NOT grow containers, only red-gem progress does.

## T2. Force-field ring (hud.js `drawFieldRing`)

Replace the plain circle so it reads as a magnetic pull field. Same signature `(g, ship, radius, t)` — deterministic in `t`, no rng, no per-frame canvas creation, no shadowBlur.

- **Two counter-rotating dashed arcs**: outer at `radius` (dash ~[14,22]), inner at `radius*0.94` (dash ~[6,26]), opposite `lineDashOffset` directions driven by `t` — reads as energized rotation, not a static circle. Reset `setLineDash([])` after.
- **Inward-drifting pull ticks**: 8 short radial line segments (length ~10px) evenly spaced around the ring, each drifting from `radius` toward `radius*0.55` on a repeating cycle (phase-offset by index, position = pure function of `t`), fading out as they approach the ship — the visual language of "things get pulled inward here". Cheap: 8 `moveTo/lineTo` strokes.
- Keep it subtle: overall alpha in the current 0.06–0.16 band (ticks may peak ~0.25 at spawn), color family `#5fe8ff`. Keep the gentle radius pulse.
- **Perf**: this draws every frame during play — ~10 strokes + dash state total; verify stress scene stays smooth (no measurable p95 regression eyeball vs v5.1's 3.8ms).

## T4. Reverse thrust on S (config.js, input.js, ship.js) — added mid-round

Julian: "add S to move backwards, might be useful."

1. `SHIP.reverseMult: 0.6` in config.js — reverse acceleration as a fraction of forward thrust (full-strength reverse would make backing off strictly dominant; 0.6 keeps commitment meaningful, tunable).
2. input.js: `S` and `↓` set `input.reverse` (mirroring the W/↑ → `input.thrust` pattern, incl. key-repeat/blur handling as done for thrust).
3. ship.js `updateShip`: when `input.reverse`, accelerate opposite the nose: `vx -= cos * a * SHIP.reverseMult * dt` (same `a = SHIP.thrust * ship.mods.engine`, so Engine upgrades and adrenaline apply). Thrust and reverse can be held together (they partially cancel — no special casing). Boost multiplies forward thrust only (boost is a forward afterburner; do not scale reverse by boost).
4. Plume: no reverse plume required; existing thrust plume keys off `input.thrust` — unchanged.
5. Tests (ship.test.js): reverse accelerates opposite the nose scaled by reverseMult; no effect when flag off; engine mod scales it; simultaneous thrust+reverse nets to (1 − reverseMult) forward.
