# V5.4 "Flight Model" — Design Spec (screen-relative crisp movement, boost sprint, wave pacing, heart cap)

**Date:** 2026-07-10, brainstormed with Julian. His verdicts: movement keys changing meaning with aim + floatiness are the gripes; he chose **crisp & tight** (Isaac/Gungeon) over inertia options. Plus: first waves are too short ("ends after 4-5 enemies wtf") and hearts must cap at 5.

Global constraints unchanged: pure-logic modules Node-clean; tuning in config.js; no per-frame shadowBlur; playable with `assets/` missing; zero deps, no build step.

## T1. Screen-relative crisp movement (config.js, input.js, ship.js, main.js, tests)

The biggest feel change since v1. Approved explicitly by Julian in brainstorm.

1. **Scheme**: W/↑ = up, S/↓ = down, A = left, D = right — screen-relative, always, regardless of aim. Mouse aims only (ship sprite still faces the cursor; `ship.angle` behavior unchanged). Diagonals normalized (|dir| = 1).
2. **Crisp**: direct velocity, not force integration. Held keys → ship velocity = dir × effectiveSpeed, with a short symmetric ramp `SHIP.rampTime: 0.05` (s to reach full speed from rest and to stop from full — implement as exponential or linear approach; must reach ≥95% within ~60ms and ≤5% within ~60ms of release; exact curve is implementer's choice, pinned by tests). No keys → velocity ramps to zero. Zero residual drift beyond the ramp.
3. **Speed model**: `effectiveSpeed = SHIP.speed × (1 + SHIP.engineSpeedBonus × (mods.engine − 1)) × adrenaline(×1.10 when active) × boost(×BOOST.speedMult while sprinting)`.
   - `SHIP.speed: 260` (replaces maxSpeed semantics).
   - `SHIP.engineSpeedBonus: 0.06` — Engine upgrade used to scale thrust (accel); accel no longer exists, so Engine becomes +6% speed per stack (cap 6 → +30% = 338 px/s). Update the Engine upgrade card copy in upgrades.js if it mentions thrust/acceleration.
4. **Removed** (delete, don't strand): `SHIP.thrust`, `SHIP.friction`, `SHIP.reverseMult`, `SHIP.strafeMult`, `SHIP.recoil` (recoil pushback is inert under direct velocity — remove its application in the fire path too), input flags `thrust/reverse/strafeLeft/strafeRight` (replaced by `up/down/left/right`).
5. **Boost = sprint** (T-integrated here, not separate): Shift while **moving** multiplies speed by `BOOST.speedMult` (1.9) in the held direction and drains the meter (`drainPerSec` × boostDrain mod); **no drain while stationary**. `BOOST.thrustMult` is removed. `ship.boosting` true only while draining (trail/afterimage keeps working). Blue-gem economy untouched.
6. **Visuals** (browser layer): thruster plume should key off movement input (any direction key held), not the old `input.thrust`; plume direction opposite the movement vector is ideal but implementer may keep it nose-based if movement-based looks wrong with the sprite — pick one, note it. Boost afterimage unchanged.
7. **Tests** (ship.test.js — REPLACE the v5.3 thrust/reverse/strafe suites): direction mapping for all 8 key combos incl. normalized diagonals; ramp timing (≥95% by 60ms, ≤5% by 60ms after release); opposite keys cancel; engine bonus math; adrenaline multiplier; sprint multiplier + drain-only-while-moving + no drain at zero input; recoil gone (fire() applies no velocity change).

## T2. Heart cap at 5 (config.js, run.js, main.js floater, tests)

1. `SHIP.maxHpCap: 5`. `applyGem` overheal grows a new container **only while `ship.maxHp < SHIP.maxHpCap`**. At the cap with full HP, a completed heart is wasted: progress consumed, return `{kind:'red', full:true}` — reinstate the v5.2-removed `full` outcome and its `FULL` floater in main.js `gemFloater`.
2. Below-max healing and partial-progress behavior unchanged. Boss heal already caps at maxHp.
3. Tests (run.test.js): grows 3→4→5; never 6; at 5-full a completed heart returns `full:true` and progress resets; heal path below max unchanged at cap; existing overheal tests updated to respect the cap.

## T3. Early wave pacing (config.js only + test updates)

`WAVE.baseBudget` 4 → **10** (wave 1 ≈ 8–10 enemies), `WAVE.budgetPerWave` 2.5 → **3**, `WAVE.spawnInterval` 1.1 → **0.9** (bigger waves shouldn't drag). Late-game terms untouched. Update any wave tests that hardcode budget expectations; formula-derived tests should pass as-is.

## Known consequences (accepted by Julian in brainstorm)

Crisp movement + screen-relative controls materially buff player dodging; combined with tripled caps this may warrant a difficulty pass — deferred to Julian's next playtest verdict. The heart cap at 5 (this spec) already pulls back part of the v5.2 power injection.
